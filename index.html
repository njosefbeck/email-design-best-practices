Email Design Best Practices

Designing and developing for e-mail can be tough. The e-mail client landscape is varied, diverse, and it can be a challenge to design in a way that looks good on all e-mail clients. As a developer, I often get design comps that don't take into account some common best practices when approaching email design. This leads to loss of time and to developer, designer, and client frustration as the e-mail needs to be reworked to fit the limitations of e-mail clients.

By following the below practices from the start a lot of friction can be avoided when designing and developing HTML email. By no means is this list exhaustive! If you'd like to add to it, feel free to get in touch at njosefbeck@gmail.com.

When designing for e-mail:

  • Don't use custom fonts.

    Stick to web-safe fonts. For safe sans-serif fonts, stick to: Arial, Arial Black, Tahoma, Trebuchet MS, and Verdana. For serif fonts, stick to: Courier, Courier New, Georgia, Times, Times New Roman.

  • If the email needs to be responsive, using solid background colors or a background patterned image work best.

  • Don't send a developer an InDesign file with the design. Do a PDF instead. Or, even better, use Sketch if you're on a Mac.

    There isn't much I can do with an InDesign file as far as extracting assets, getting correct typography sizes, etc. Print does not equal web!

  • Send full-size versions of all image and icon assets (zipped up, outside of a PSD).
